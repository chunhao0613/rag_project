## 03（4~6h）：容器化基線

### 這階段做了什麼
- 建立 [Dockerfile](../Dockerfile)
- 建立 [.dockerignore](../.dockerignore)
- 驗證映像可 build、可 run

### 如何撰寫這段 code（code 解釋）

#### 1) [Dockerfile](../Dockerfile)

```dockerfile
FROM python:3.11-slim

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1

WORKDIR /app

RUN apt-get update \
    && apt-get install -y --no-install-recommends curl make \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

COPY . ./

RUN mkdir -p data/uploads

EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=5s --start-period=30s --retries=3 \
    CMD curl -fsS http://localhost:8000/_stcore/health || exit 1

CMD ["python", "-m", "streamlit", "run", "app.py", "--server.port", "8000", "--server.address", "0.0.0.0"]
```

重點設計：
- `python:3.11-slim`：在相容性與體積間取平衡。
- `COPY requirements.txt` 先於 `COPY .`：最大化 layer cache 命中率。
- `HEALTHCHECK`：容器不只「活著」，還要能回應服務。
- `make` 一併安裝：後續 05 階段可直接在容器執行 `make check`。

#### 2) [.dockerignore](../.dockerignore)

核心策略是「不把不該進映像的內容放進 build context」：
- 快取/暫存：`__pycache__/`, `.pytest_cache/`, `.ruff_cache/`
- 機密：`.env`
- 不需封裝的資料：`data/chroma_db*`, `data/uploads/`
- 開發文件：`docs/`

這樣可以降低 build context 大小、縮短建置時間，並避免敏感資訊進映像。

### 驗證方式
- `docker build` 成功
- `docker run` 可啟動並提供服務

## 04（6~8h）：本機編排與健康檢查

### 這階段做了什麼
- 建立 [docker-compose.yml](../docker-compose.yml)
- 補齊服務健康檢查路徑
- 驗證 `docker compose up -d --build` 可啟動
- 驗證 `docker compose ps` 顯示 `healthy`

### 如何撰寫這段 code（code 解釋）

#### [docker-compose.yml](../docker-compose.yml)

```yaml
services:
  rag-app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: rag_app
    ports:
      - "8000:8000"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000"]
      interval: 30s
      timeout: 10s
      retries: 3
    volumes:
      - ./data:/app/data
    environment:
      - PYTHONUNBUFFERED=1
      - TZ=Asia/Taipei
```

重點設計：
- `ports: 8000:8000`：對齊 Streamlit 實際監聽埠。
- `healthcheck`：以 HTTP 200 作最小可用判定。
- `volumes`：保留向量庫/上傳資料，容器重建後仍可使用。
- `PYTHONUNBUFFERED=1`：即時 log，排障更快。

### readiness / liveness 面試要點（此階段先能清楚解釋）
- **Docker healthcheck**：容器層級的健康訊號。
- **Kubernetes readinessProbe**：決定 Pod 是否可接收流量。
- **Kubernetes livenessProbe**：決定容器是否該被重啟。

本階段先在 Compose 完成「最小健康檢查」，後續到 K8s（08 階段）再分拆 `readiness` / `liveness` 行為。

### 驗證方式
- `docker compose up -d --build` 成功
- `docker compose ps` 顯示 `Up ... (healthy)`
- 重啟後可恢復（`docker compose down && docker compose up -d`）

## 08（14~16h）：探針與資源治理

### 這階段做了什麼
- 在 [k8s/base/deployment.yaml](../k8s/base/deployment.yaml) 加入 `readinessProbe`
- 在 [k8s/base/deployment.yaml](../k8s/base/deployment.yaml) 加入 `livenessProbe`
- 在 [k8s/base/deployment.yaml](../k8s/base/deployment.yaml) 加入 `requests/limits`

### 如何撰寫這段 code（code 解釋）

#### 1) Probe 設計
- `readinessProbe`：`/_stcore/health`，失敗時「停止接流量」但不立即重啟
- `livenessProbe`：同一路徑，若連續失敗則由 kubelet 重啟容器

這樣可以區分「暫時不可服務」與「程序失活需重啟」。

#### 2) 資源治理
- `requests`: `cpu: 250m`, `memory: 512Mi`
- `limits`: `cpu: 1`, `memory: 1Gi`

目的：
- 讓 scheduler 知道最小資源需求
- 防止單一工作負載吃光節點資源

### 驗證方式
- 觀察 `kubectl describe pod` 的 probe 狀態與事件
- 觀察 OOM / throttling 事件是否可被限制與追蹤

## 10（18~20h）：YAML 版回滾演練

### 這階段做了什麼
- 在 minikube 套用 base manifests，建立穩定基線
- 故意把 Deployment image 升級成不存在標籤（壞版本）
- 使用 `kubectl rollout undo` 回復到上一個穩定 revision
- 保留 rollout history 作為升級/降版證據

### 如何撰寫這段 code（code 解釋）

#### 1) 升級（故障注入）
演練命令核心是：
- `kubectl -n rag-mvp set image deployment/rag-app rag-app=rag_project-rag-app:does-not-exist`

目的不是真的上壞版，而是可控地製造 `ImagePullBackOff`，驗證回滾流程可用。

#### 2) 回滾（復原路徑）
回滾命令核心是：
- `kubectl -n rag-mvp rollout undo deployment/rag-app`
- `kubectl -n rag-mvp rollout status deployment/rag-app`

回滾後確認 image 已回到 `rag_project-rag-app:latest`。

#### 3) 歷史追蹤
- `kubectl -n rag-mvp rollout history deployment/rag-app`

本次演練可看到 revision 變化（壞版 revision 與回滾後 revision）。

### 實測證據（本次環境）
- `deployment.apps/rag-app rolled back`
- `deployment "rag-app" successfully rolled out`
- rollback 後 image：`rag_project-rag-app:latest`
- rollout history 顯示壞版與回滾 revision

### 驗證方式
- 壞版升級可被偵測（rollout status 無法成功）
- 執行 undo 後回到可用狀態
- history 可追到變更原因與 revision

## 15（28~30h）：GitLab CI 治理閘道

### 這階段做了什麼
- 新增 [/.gitlab-ci.yml](../.gitlab-ci.yml)
- 拆出三段式 gate：`validate`、`quality`、`security`
- 加入 `trigger_jenkins` 作為後續交付橋接

### 如何撰寫這段 code（code 解說）

#### 1) validate gate
- 執行 `python -m compileall app.py core services`
- 執行 `ruff check core services tests app.py`

用途：先抓語法與靜態錯誤，避免壞碼進入更後面的流程。

#### 2) quality gate
- 執行 `make test`

用途：保證核心行為（文件處理、向量化、模型選擇）仍可運作。

#### 3) security gate
- 安裝並執行 `pip-audit`
- 安裝並執行 `bandit`

用途：把依賴漏洞與常見 Python 安全問題先擋下。

### 概念、技術與專有名詞

- **Pipeline**：由多個 job 組成的自動化流程
- **Stage**：job 的階段順序
- **Gate**：通過門檻，不通過就不往下
- **pip-audit**：Python 依賴漏洞掃描
- **Bandit**：Python 原始碼安全掃描

### 實作檔案重點

```yaml
stages:
  - validate
  - quality
  - security
  - trigger
```

```yaml
validate:
  script:
    - python -m compileall app.py core services
    - ruff check core services tests app.py
```

```yaml
quality:
  script:
    - make test
```

```yaml
security:
  script:
    - pip install pip-audit bandit
    - pip-audit -r requirements.txt
    - bandit -q -r core services app.py
```

解說重點：

- `validate` 先擋語法與靜態問題。
- `quality` 再驗證核心行為。
- `security` 最後補上漏洞與風險掃描。

### 驗證方式
- 若任一 job fail，後續 stage 不會執行。
- 這就符合「任一 gate fail 不得部署」的治理要求。

## 16（30~32h）：GitLab 安全強化

- 讓高風險依賴或危險寫法先被攔住

### 如何撰寫這段 code（code 解說）

#### 1) 依賴掃描
- `pip-audit -r requirements.txt`

用途：查看 Python 套件是否有已知 CVE 或風險版本。

#### 2) 程式碼安全掃描

### 概念、技術與專有名詞

- **Dependency scanning**：掃依賴漏洞
- **Static security scanning**：掃原始碼風險
- **Policy**：遇到高風險結果就阻擋

### 驗證方式
- 只要掃描出高風險問題，pipeline 就應失敗。
- 這讓安全從「上線後才補」變成「上線前先擋」。

## 17（32~34h）：GitLab 觸發 Jenkins

### 這階段做了什麼
- 新增 `trigger_jenkins` job
- 只有當前面三個 gate 全綠、且在 `main` 分支時才觸發 Jenkins
- 以 webhook / token / buildWithParameters 方式交接下一段交付

### 如何撰寫這段 code（code 解說）

#### 1) 前置條件
- `needs: [validate, quality, security]`

用途：明確要求只有前面三關都過，才允許進入 Jenkins 觸發。

#### 2) Jenkins 觸發
- 使用 `curl` 呼叫 Jenkins 的 `buildWithParameters`
- 帶入 `branch` 與 `commit`，方便 Jenkins 追溯來源

#### 3) 安全保護
- 若 `JENKINS_URL/JENKINS_JOB/JENKINS_TOKEN` 未設定，job 直接跳過

### 概念、技術與專有名詞

- **Webhook**：事件觸發的 HTTP 呼叫
- **Parameterized build**：帶參數觸發 Jenkins job
- **Token**：簡易授權或觸發金鑰

### 實作檔案重點

```yaml
trigger_jenkins:
  stage: trigger
  needs:
    - validate
    - quality
    - security
  script:
    - |
      if [ -z "$JENKINS_URL" ] || [ -z "$JENKINS_JOB" ] || [ -z "$JENKINS_TOKEN" ]; then
        echo "Skip Jenkins trigger: JENKINS_URL/JENKINS_JOB/JENKINS_TOKEN not set."
        exit 0
      fi
    - |
      curl -fsS -X POST "${JENKINS_URL%/}/job/${JENKINS_JOB}/buildWithParameters?token=${JENKINS_TOKEN}&branch=${CI_COMMIT_REF_NAME}&commit=${CI_COMMIT_SHA}"
```

解說重點：

- `needs` 是硬性依賴，避免繞過 gate。
- `buildWithParameters` 把 GitLab 的結果傳給 Jenkins。
- 這一段還不是 Jenkinsfile，本質上是 GitLab 到 Jenkins 的交接橋。

### 驗證方式
- 只有 `main` 且前面 gate 全綠，才會觸發 Jenkins。
- 這樣可保證「gate 全綠才觸發 Jenkins」。

## 面試官問答（03~17 常見追問）

### Q1：為什麼先做 Compose，不直接上 Kubernetes？
A：先在本機把 build、啟動、健康檢查與重啟流程穩定化，能更快縮小問題範圍。這是低成本建立「可重現部署單位」的方式，再上 K8s 才不會把問題放大。

### Q2：你怎麼區分 Docker healthcheck 與 K8s readiness/liveness？
A：Docker healthcheck 是容器層訊號；K8s readiness 決定是否接流量、liveness 決定是否重啟。04 階段先做健康判定，08 階段再做 probe 分工。

### Q3：為什麼 05 階段不追高 coverage？
A：MVP 優先是建立可執行品質 gate，先覆蓋核心失敗路徑，確保低品質變更被擋下；等交付鏈穩定後，再逐步擴覆蓋率與更細緻測試。

### Q4：這套品質入口如何接到 GitLab CI？
A：直接以 `make check` 當 pipeline 第一關。fail 即阻擋後續 build/deploy，避免壞版本流入交付鏈。

### Q5：你怎麼證明結果可重現？
A：指令固定、環境固定（容器），驗收固定（healthy + `make check`）。同一份 repo 在同一流程可得到一致結果。

### Q6：為什麼 Secret 要用 template，不直接放真值？
A：Git 倉庫是長期可追溯資產，敏感資料不應進版本歷史。模板保留欄位結構，實際值由部署環境注入，符合最小暴露原則。

### Q7：readiness 與 liveness 為什麼要同時存在？
A：兩者責任不同。readiness 控制流量門禁，liveness 控制重啟決策。若混用，會造成不必要重啟或錯誤接流量。

### Q8：為什麼先用 PVC 而不是物件儲存？
A：MVP 先解決「Pod 重建資料不丟」這個最低可行目標。PVC 對單一工作負載最直接，等後續再評估物件儲存與資料分層。

### Q9：如果要擴到多副本，PVC 會不會卡住？
A：RWO 在多副本會有限制，這是有意識的 trade-off。現階段先穩定單副本與資料正確性，下一階段再評估 RWX 或外部向量服務。

### Q10：YAML 版回滾為什麼重要？
A：因為 SRE 的重點不只「能部署」，而是「故障時可快速回復」。透過 rollout history + undo，我能把回復流程標準化，直接對應 MTTR 目標。

### Q11：你怎麼證明回滾不是口說？
A：我做了故障注入（錯誤 image tag），再執行 rollback，最後用 rollout status / history / image 值回查，形成可審計證據鏈。

### Q12：為什麼先 Helm 初始化，不直接改到完整模板？
A：先建立 chart 邊界，確保工具鏈可運作（lint、template），再逐步把現有 YAML 模板化。這樣能避免一次改動過大，降低排錯成本。

### Q13：values-dev / staging / prod 的價值是什麼？
A：同一套模板可覆蓋多環境，降低重複 YAML 與配置漂移風險；差異集中在 values，審核也更清楚。

### Q14：為什麼 Helm upgrade 會因 PVC 失敗？
A：因為 PVC 容量變更受 storage class 能力限制。這正是實務上要先驗證儲存策略的原因；遇到失敗時應保留 history 並可快速 rollback。

### Q15：Helm rollback 跟 kubectl rollout undo 怎麼分工？
A：`kubectl rollout undo` 是針對 workload（如 Deployment）回退；`helm rollback` 是整個 release 維度回退（含多資源模板），治理粒度更完整。

### Q16：為什麼 GitLab 要先做治理閘道，而不是直接交給 Jenkins？
A：因為 GitLab 的角色是准入治理，應該在最上游先擋掉壞碼與高風險變更，避免後面 build/deploy 浪費成本。Jenkins 是執行器，不是第一道判斷者。

### Q17：validate、quality、security 三段式的分工是什麼？
A：validate 擋語法與靜態錯誤，quality 驗證核心行為，security 擋依賴漏洞與原始碼風險。三個 gate 各自負責不同風險層級。

### Q18：為什麼用 pip-audit 和 bandit？
A：pip-audit 專注套件漏洞，bandit 專注 Python 原始碼安全模式。兩個一起用能同時補住 supply chain 與程式碼層風險。

### Q19：GitLab 觸發 Jenkins 時，為什麼還要帶 branch 和 commit？
A：這樣 Jenkins 能知道是哪個分支與哪個 commit 觸發，方便追溯、重現與對帳。交付鏈的每一跳都應保留上下文。

### Q20：如果 Jenkins 端沒設定好，這個 job 怎麼辦？
A：目前設計成若 `JENKINS_URL/JENKINS_JOB/JENKINS_TOKEN` 沒有設定就跳過，避免因環境未完成而把 GitLab pipeline 卡死；真正上線時再把這些變數納入受保護設定。

## 18（34~36h）：Jenkins 前半管線

### 這階段做了什麼
- 新增 [Jenkinsfile](../Jenkinsfile)
- 把 CI/CD 的前半段拆成 `Checkout`、`Build`、`Test`、`Scan`
- 建立可審計的 image tag 與 commit 追蹤

### 如何撰寫這段 code（code 解說）

#### 1) Checkout
- 先 `checkout scm`
- 再用 `git rev-parse --short HEAD` 取得短 commit id

用途：固定來源版本，讓後面的 build 與部署都可以回查到是哪個 commit。

#### 2) Build
- `docker build -t rag_project-rag-app:<build>-<sha> .`

用途：把目前版本打包成 image，作為後續 push / deploy 的共同工件。

#### 3) Test
- 直接跑 `make check`

用途：沿用 05 階段建立的品質入口，避免在 Jenkins 裡又寫一套新的測試命令。

#### 4) Scan
- 安裝 `pip-audit` 與 `bandit`
- 執行依賴與原始碼安全掃描

用途：把 supply chain 與 Python 程式碼風險一起擋掉。

### 概念、技術與專有名詞

- **Pipeline**：由多個 stage 組成的自動化流程
- **Checkout**：抓取源碼
- **Artifact**：產出的工件，這裡是 image
- **Scan**：安全檢查
- **Log evidence**：把構建與掃描結果保留下來，方便追蹤

### 實作檔案重點

```groovy
stage('Checkout') {
  steps {
    checkout scm
    script {
      env.GIT_SHA_SHORT = sh(returnStdout: true, script: 'git rev-parse --short HEAD').trim()
      env.IMAGE_TAG = "${env.BUILD_NUMBER}-${env.GIT_SHA_SHORT}"
    }
  }
}
```

```groovy
stage('Build') {
  steps {
    sh '''
      set -euo pipefail
      docker build -t "${LOCAL_IMAGE}" .
      docker image inspect "${LOCAL_IMAGE}" >/dev/null
    '''
  }
}
```

```groovy
stage('Test') {
  steps {
    sh 'make check'
  }
}
```

```groovy
stage('Scan') {
  steps {
    sh '''
      set -euo pipefail
      python -m pip install --upgrade pip
      python -m pip install pip-audit bandit
      pip-audit -r requirements.txt
      bandit -q -r core services app.py
    '''
  }
}
```

解說重點：

- `checkout scm`：從 Jenkins 所對應的 repo 取出當前版本。
- `git rev-parse --short HEAD`：把 commit 變成短 id，方便命名與追蹤。
- `docker build`：把源碼轉成 image。
- `docker image inspect`：確認 image 真正存在，避免只看 build log。
- `make check`：把 lint/test 串成單一品質入口。
- `pip-audit` / `bandit`：分別掃依賴漏洞與程式碼風險。

### 驗證方式
- 前半段可穩定產出 image
- `make check` 可直接沿用
- 掃描結果可在 Jenkins log 中追蹤

## 19（36~38h）：Jenkins 後半管線

### 這階段做了什麼
- 新增 `Push`、`Deploy`、`Smoke Test`
- 若有 registry 資訊，就推送 image
- 使用 Helm `upgrade --install` 部署到 `rag-helm`
- 以 rollout status 與 curl smoke test 驗證健康檢查

### 如何撰寫這段 code（code 解說）

#### 1) Push
- 若有 `DOCKER_REGISTRY` 與認證，就把 image push 上去

用途：把本機 image 變成共享工件，讓部署環境可以拉取。

#### 2) Deploy
- 使用 `helm upgrade --install`
- 套用 `helm/rag-app` 與 `values-prod.yaml`
- 部署前先記錄上一個 deployed revision

用途：讓安裝與升級用同一條命令，並為後續回滾留下目標版本。

#### 3) Smoke Test
- 先 `kubectl rollout status`
- 再用臨時 `curlimages/curl` Pod 連到 `http://rag-app/_stcore/health`

用途：只驗證最小可用路徑，確認服務真的對外健康。

### 概念、技術與專有名詞

- **Push**：推送映像到 registry
- **Deploy**：把版本交付到叢集
- **Smoke test**：最小健康驗證
- **Helm release**：Helm 管理的部署單位
- **Rollout status**：確認部署完成與否

### 實作檔案重點

```groovy
stage('Push') {
  when {
    expression { return env.DOCKER_REGISTRY?.trim() }
  }
}
```

```groovy
stage('Deploy') {
  steps {
    sh '''
      previous_revision="$(helm history "${HELM_RELEASE}" -n "${HELM_NAMESPACE}" 2>/dev/null | awk '$2=="deployed"{rev=$1} END{print rev}')"
      helm upgrade --install "${HELM_RELEASE}" "${HELM_CHART}" \
        -n "${HELM_NAMESPACE}" --create-namespace \
        -f "${HELM_VALUES}" \
        --set image.repository="${DEPLOY_IMAGE_REPO}" \
        --set image.tag="${DEPLOY_IMAGE_TAG}"
    '''
  }
}
```

```groovy
stage('Smoke Test') {
  steps {
    sh '''
      kubectl -n "${HELM_NAMESPACE}" rollout status deployment/"${HELM_RELEASE}" --timeout="${ROLLOUT_TIMEOUT}"
      kubectl -n "${HELM_NAMESPACE}" run smoke-${BUILD_NUMBER} --image=curlimages/curl:8.9.1 --restart=Never --rm -i -- curl -fsS "http://${HELM_RELEASE}${SMOKE_PATH}"
    '''
  }
}
```

解說重點：

- `when` 讓推送動作只在有 registry 參數時執行。
- `helm upgrade --install` 是 Helm 的標準交付命令。
- `previous_revision` 是 rollback 的回復點。
- `kubectl rollout status` 與 `curl smoke` 一起使用，才能確認服務真的可用。

### 驗證方式
- 有 registry 時可 push
- 部署後能看到 rollout 成功
- smoke test 能打到 health endpoint

## 20（38~40h）：Jenkins 自動回滾

### 這階段做了什麼
- 在 Jenkinsfile 的 `post { failure { ... } }` 裡實作自動回滾
- 若 deploy 或 smoke test 失敗，就讀取上一個穩定 revision
- 有可回滾版本時，執行 `helm rollback`
- 同時保留 `helm history` 與 smoke log 作為證據

### 如何撰寫這段 code（code 解說）

#### 1) 先記錄回滾目標
- 在 `Deploy` 階段前先查 `helm history`
- 找出最後一個 `deployed` 的 revision

用途：避免失敗後才臨時猜要回到哪版。

#### 2) 失敗後觸發 rollback
- `post { failure { ... } }`
- 讀 `.jenkins_previous_revision`
- 執行 `helm rollback`

用途：把補救流程標準化，減少人工介入時間。

#### 3) 留存證據
- `helm history` 再輸出一次
- `archiveArtifacts` 保存 `.jenkins_*` 檔案

用途：回滾不只是恢復服務，還要能回溯「發生了什麼、回到哪裡」。

### 概念、技術與專有名詞

- **Failure hook**：失敗後自動執行的邏輯
- **Rollback target**：要退回的穩定版
- **Revision**：Helm release 版本號
- **Artifact**：保留下來的 log 或狀態檔

### 實作檔案重點

```groovy
post {
  failure {
    sh '''
      previous_revision="$(cat .jenkins_previous_revision 2>/dev/null || true)"
      if [ -n "${previous_revision}" ]; then
        helm rollback "${HELM_RELEASE}" "${previous_revision}" -n "${HELM_NAMESPACE}" --wait --timeout="${ROLLOUT_TIMEOUT}"
      fi
    '''
  }
  always {
    archiveArtifacts artifacts: '.jenkins_*', allowEmptyArchive: true
  }
}
```

解說重點：

- `post.failure` 是 rollback 的執行點。
- `cat .jenkins_previous_revision` 讓回滾目標可追溯。
- `helm rollback` 直接回到穩定版。
- `archiveArtifacts` 把狀態與 log 保留下來，讓排障可重現。

### 驗證方式
- deploy / smoke 失敗時可自動回復
- 回滾後可再查 `helm history`
- 留有對應 log 與狀態檔

## 面試官問答（18~20 常見追問）

### Q21：為什麼 Jenkins 前半管線要先 checkout 再 build，再 test，最後 scan？
A：先固定來源版本，再產出工件，接著驗證核心行為，最後做安全掃描。這樣最早可以在低成本階段擋掉問題，也能保留完整證據鏈。

### Q22：為什麼後半管線要用 Helm 而不是直接 kubectl apply？
A：因為 Helm 同時管理模板、release 版本與 rollback。部署不只要把資源送進去，還要能追蹤版本與快速回復。

### Q23：smoke test 為什麼只驗 health endpoint？
A：smoke test 的目的不是測完整功能，而是確認最小可用路徑。健康檢查通過代表服務至少能接收請求並回應。

### Q24：rollback 為什麼要放在 `post failure` 而不是獨立 stage？
A：rollback 是補救動作，只應在失敗後執行。放在 `post failure` 可確保不污染正常流程，也能對所有失敗來源統一處理。

### Q25：如果沒有前一版可回滾怎麼辦？
A：就跳過 rollback，因為第一次 install 沒有上一版可回。這也是為什麼 deploy 前要先查 `helm history` 並把 revision 記錄下來。
